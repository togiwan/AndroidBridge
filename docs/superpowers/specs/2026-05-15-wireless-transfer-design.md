# AndroidBridge Wireless Transfer Design

## Goal

Add wireless transfer to AndroidBridge without weakening the app's current USB experience.

Wireless transfer should serve two kinds of users:

- Users who want the easiest possible app-free transfer flow.
- Users who want the full Android file browser experience without a cable.

The first wireless release should make Browser Transfer the default experience and expose ADB Wireless as an advanced full-access mode.

## Product Shape

AndroidBridge will have two primary transfer tabs:

- USB Transfer
- Wireless Transfer

USB Transfer keeps the current ADB-over-USB behavior.

Wireless Transfer contains:

- Browser Transfer
- ADB Wireless

Browser Transfer is the default wireless panel. ADB Wireless is available for users who need USB-like folder browsing, preview, upload, download, and full ADB file access.

## Browser Transfer

Browser Transfer lets a user move files between a Mac and an Android phone over the local network with no Android app install and no ADB requirement.

The Mac app starts a temporary local web server. The user opens the session on the Android phone by scanning a QR code or typing the shown URL. The phone uses its normal browser.

### Session Start

When the user starts Browser Transfer:

1. AndroidBridge asks for a Receive Folder on the Mac.
2. AndroidBridge starts a local web server.
3. AndroidBridge creates a random session token and PIN.
4. AndroidBridge shows a QR code, local URL, PIN, receive folder, and Stop Session control.

The session should be explicit and temporary. The server runs only while the session is active.

### Phone To Mac

The phone web page includes a Send to Mac area.

The user selects one or more files with the browser file picker. AndroidBridge saves uploads automatically into the selected Receive Folder. The Mac app shows active upload progress, completed files, failures, and cancellation where possible.

The first version should not require per-file approval on the Mac. The Receive Folder choice is the approval boundary.

### Mac To Phone

The Mac app lets the user add downloadable items while the session is running:

- Add Files
- Add Folder
- Remove selected item
- Clear shared items

The phone web page includes a Get from Mac area that lists the currently shared items.

Files download directly. Folders download as ZIP archives generated by the Mac app. The first version can generate ZIPs on demand and report progress as indeterminate when exact folder size is not available.

### Browser Page

The phone browser UI should be simple:

- Session name or AndroidBridge title
- Send to Mac
- Get from Mac
- Upload/download progress
- Clear success and failure states

The page should not expose Mac paths. It should show friendly item names, sizes when available, and whether an item is a file or folder archive.

### Security

Browser Transfer should be local, temporary, and token-gated.

Requirements:

- The web server starts only after the user starts a session.
- Session URLs include a random token.
- The PIN is required before the phone can transfer files.
- Stopping the session invalidates all URLs and tokens.
- The server binds to the local network only.
- The receive folder is the only automatic upload destination.
- Shared downloads are limited to items explicitly added by the user.
- The Mac UI clearly shows when a wireless session is active.

Nice-to-have safeguards:

- Auto-stop after an idle timeout.
- Auto-stop after app quit or sleep.
- Copy URL button.
- Regenerate token button.
- Warning if no local network address is available.

## ADB Wireless

ADB Wireless is the full-access wireless mode. It should reuse the current ADB file browser behavior after a device is connected.

This mode is for users who want wireless access to:

- Browse Android folders.
- Download Android files and folders.
- Upload Mac files and folders.
- Open Android files through temporary local preview.
- Use existing progress, cancellation, and helpful ADB errors.

### Discovery First

The user experience should prefer automatic discovery before manual setup.

The ADB Wireless panel should:

1. Scan for Wireless Debugging services on the local network.
2. Show discovered devices or services when available.
3. Guide pairing using the pairing code shown on Android.
4. Auto-connect after successful pairing when possible.
5. Fall back to manual entry when discovery fails.

Manual setup should be secondary, not the main flow.

### Guided Pairing

AndroidBridge should guide the user through Android's Wireless Debugging flow:

1. Open Developer Options on the Android phone.
2. Enable Wireless debugging.
3. Choose Pair device with pairing code.
4. Enter the pairing code in AndroidBridge.
5. Let AndroidBridge pair and connect.

If discovery finds the pairing address, the user should only need to enter the pairing code. If discovery does not work, the manual fallback asks for:

- Pairing address
- Pairing code
- Connection address

### ADB Commands

ADB Wireless uses standard ADB commands:

- `adb pair <ip>:<pairing-port> <code>`
- `adb connect <ip>:<connection-port>`
- `adb devices`
- Existing `adb -s <deviceID> ...` file commands after connection

After connection, the current ADB-backed file browser can operate on the wireless device like a USB device.

### Errors

ADB Wireless should provide helpful errors for:

- Pairing timeout
- Wrong pairing code
- Phone and Mac not on the same network
- Wireless debugging disabled
- Connect failed after pairing
- Device unauthorized, offline, or disconnected
- ADB not installed

## Architecture

Wireless Transfer should add new capabilities without mixing browser-server concerns into the current ADB client.

Suggested core units:

- `WirelessTransferSession`
- `WirelessTransferServer`
- `WirelessTransferToken`
- `SharedDownloadItem`
- `ReceivedUpload`
- `ADBWirelessClient`
- `ADBWirelessDiscovery`

Suggested app/store shape:

- Keep existing USB ADB browsing behavior intact.
- Add a transfer mode enum for USB and Wireless.
- Add a wireless submode enum for Browser Transfer and ADB Wireless.
- Move browser session state into a focused model rather than expanding `AndroidBridgeStore` indefinitely.
- Reuse current transfer progress display patterns where possible.

The local web server should be isolated behind a small interface so it can be tested independently from SwiftUI.

## Testing

Core tests should cover:

- Token generation and validation.
- Session start and stop state transitions.
- Shared item list add/remove/clear behavior.
- Receive folder path handling.
- Upload destination collision behavior.
- Folder-to-ZIP naming behavior.
- ADB Wireless command construction.
- ADB Wireless output/error parsing.

Manual verification should cover:

- Phone to Mac upload from Android Chrome.
- Mac to Phone file download.
- Mac to Phone folder download as ZIP.
- Session stop invalidates phone page.
- Wrong PIN is rejected.
- ADB Wireless discovery success path when available.
- ADB Wireless manual fallback.
- Existing USB Transfer still works.

## MVP Scope

The first implementation should include:

- Main USB Transfer and Wireless Transfer tabs.
- Browser Transfer as the default wireless view.
- Start/stop browser session.
- Receive Folder selection.
- QR code and URL display.
- Token and PIN protection.
- Phone to Mac uploads.
- Mac to Phone file downloads.
- Mac to Phone folder downloads as ZIP.
- Add Files, Add Folder, Remove, and Clear controls.
- Transfer status in the Mac UI.
- ADB Wireless discovery attempt.
- Guided ADB Wireless pairing.
- Manual ADB Wireless fallback.
- Reuse of existing ADB browser after wireless connection.

Out of scope for the first implementation:

- Native Android companion app.
- Cloud relay transfer.
- Internet transfer outside the local network.
- Full recursive folder progress for Browser Transfer ZIP creation.
- Perfect ADB discovery on every Android device.
- Persistent trusted-device management.

## Open Product Decisions

Resolved:

- Browser Transfer is the default wireless experience.
- ADB Wireless remains available for full file access.
- Browser Transfer includes both Phone to Mac and Mac to Phone.
- Mac to Phone shared items can be added while the session is running.
- Phone to Mac uploads save automatically to the selected Receive Folder.
- ADB Wireless should try automatic discovery before showing manual fields.

Remaining:

- Exact idle timeout duration.
- Whether PIN entry is required once per browser session or for every page load.
- Whether duplicate uploaded filenames should overwrite, auto-rename, or ask on Mac.
- Whether QR generation uses an in-repo implementation or a Swift package dependency.
- Which embedded HTTP server implementation to use.

## Recommended Next Step

Write an implementation plan that delivers Browser Transfer first, then layers ADB Wireless on top.

Browser Transfer should be implemented in vertical slices:

1. Session state and local server skeleton.
2. Mac UI for start/stop, QR/URL, and receive folder.
3. Phone web UI.
4. Phone to Mac upload.
5. Mac to Phone downloads.
6. Folder ZIP support.
7. Security hardening and timeout.

ADB Wireless should follow once Browser Transfer is stable:

1. ADB pair/connect wrappers.
2. Discovery attempt.
3. Guided pairing UI.
4. Manual fallback.
5. Integration with the existing device list and file browser.
